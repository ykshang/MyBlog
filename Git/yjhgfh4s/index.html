常见的概念

约 3037 字大约 10 分钟

2025-11-11

核心概念

1、仓库 (Repository)

什么是仓库？仓库就是存放文件的地方，它可以是本地的，也可以是远程的。仓库里组织代码的单位叫做提交（Commit）。

基于仓库的位置，一般分为：

  • 本地仓库：存储在本地计算机上的 Git 仓库
  • 远程仓库：存储在服务器上的 Git 仓库（如 GitHub、GitLab）

2、提交 (Commit)

提交是 Git 中用来组织文件内容的最小单位，将暂存区的变更永久保存到本地仓库，形成版本快照。每个提交都有一些关键信息，包括提交哈希值、提交作者、提交时间、提交描述、前一个提交 ID 等，方便后续的版本管理和回滚操作。

# 每次提交都会生成一个唯一的哈希值
git commit -m "feat: add user authentication"
# 提交哈希示例：a1b2c3d4e5f6...

3、提交历史 (Commit History)

提交历史是指仓库中所有提交的记录，按照时间顺序排列。每个提交都有一个唯一的哈希值，用于标识该提交。通过查看提交历史，我们可以了解项目的开发过程、功能实现、Bug 修复等。

Git 在操作工作目录时，就是基于提交历史来进行的。每次操作，都需要先获取两种状态之间的对应的提交历史差异，逐个应用到当前工作目录，最终更新到最新状态。

4、分支 (Branch)

分支是 Git 中用来并行开发不同功能的机制。每个分支都是一个独立的线，用于开发和维护不同的功能。通过分支，你可以在不影响主分支的情况下，独立地开发新功能或修复 Bug。

一般来说，我们会按照用途来命名分支，例如：

  • main 或 master：主分支，用于稳定版本
  • develop：开发分支，用于整合新功能
  • feature/：功能分支，用于开发新功能
  • hotfix/：修复分支，用于紧急修复 bug
  • release/：发布分支，用于版本发布准备

分支的实质是一个指针, 指向某一个提交

5、标签 (Tag)

标签也是一种指针，它用来标记某一个提交，通常用来标记发布版本或重要的里程碑。

6、指针 (Pointer)

在 Git 中，指针是一个很重要的概念，像分支、标签、HEAD 等，他们的实质是一个指向某个提交的指针。

为什么会有指针的概念？

我们也可以理解为指针是某一个提交对应的 commit id 的别名，指向的是 某一种工作目录的状态。有了指针，可以大大的简化我们的操作，不需要指定具体的提交哈希值。

举几个例子来描述一下他如何工作：

  1. 我们要切换分支，Git 会比较 HEAD 和目标分支，这两个指针之间的相对历史，根据历史，更新当前工作目录的代码到目标分支的状态。
  2. 我们要拉取代码，Git 比较的是本地分支（也有可能是 HEAD）跟远端分支之间的提交历史差异，先通过 fetch 来将远端分支的最新提交历史拉取到本地，然后再根据两个指针之间的差异来更新代码到最新状态。

7、钩子 (Hooks)

Git 中的钩子 (Hooks) 是一种在 Git 操作过程中自动触发的脚本，用于执行自定义的操作。钩子分为客户端钩子和服务端钩子。

客户端钩子 (Client-side Hooks)

运行在本地计算机上，用于在提交或推送代码前执行自定义操作。常用的客户端钩子有 pre-commit, pre-push。

服务端钩子 (Server-side Hooks)

运行在服务器上，用于在接收代码前执行自定义操作。常用的服务端钩子有 pre-receive, post-receive。

三个重要区域

工作目录 (Working Directory)

工作目录下的文件，是我们实际操作的文件，我们可以在工作目录下修改、删除、添加文件，也是 Git 版本控制的对象。

暂存区 (Staging Area)

暂存区可以理解为一个缓存区域，存放了我们准备提交的修改的文件和内容。

仓库 (Repository)

仓库里是保存所有提交的文件快照，每个提交都有一个唯一的哈希值，用于标识该提交。

文件状态生命周期

  • 未跟踪 (Untracked) ：一般新增的文件，未被提交过、也未被添加到暂存区，因此未被 Git 跟踪，未形成版本控制。他的状态是未跟踪。
  • 已暂存 (Staged) ：不管是新增、修改还是删除，涉及的文件被添加到暂存区，准备提交的状态叫做已暂存。
  • 已提交 (Committed) ：改动的文件已被提交到本地仓库，形成版本快照。
  • 已修改 (Modified) ：已经被 git 版本控制文件，被修改了。
  • 未暂存的修改 (Unstaged) ：已经被 git 版本控制文件，被修改了，但未被添加到暂存区。

操作概念

1、检出 (Checkout)

检出是将仓库切换到某个状态，一般是切换到某个分支或者某个提交，并将工作区恢复到对应的状态。它的功能过于强大和复杂。

现在一般推荐使用git fetch 来同步历史，使用git switch 来切换分支，使用 git restore 来恢复文件。

2、同步（Fetch）

同步指的是仓库之间的提交历史同步，一般是将远程仓库的提交拉取到本地仓库，使本地仓库与远程仓库保持同步。

同步只下载提交历史，同时更新本地仓库中的远端分支对应的指针。

3、拉取 (Pull)

拉取是将远程仓库的提交拉取到本地仓库，使本地仓库与远程仓库保持同步。

拉取其实就是 fetch + merge 的语法糖

拉取的实质就是先通过 fetch 来将远端分支的最新提交历史拉取到本地，然后使用 merge 的方式将本地分支合并到远程分支。当然我们可以指定参数 --rebase 来使用变基的方式合并两个分支。

merge 是拉取的默认行为，我们可以使用 git config --global pull.rebase true 来将拉取的默认行为设置为 rebase。

4、推送 (Push)

推送是将本地仓库的提交推送到远程仓库，使远程仓库与本地仓库保持同步。推送的过程中，远端仓库需要根据你推送的历史，来确定相对历史的差异，并根据差异来确定能否更新成功：

  • 成功的话，远端仓库会同步历史，并更新对应的指针，指向你推送的最新提交
  • 失败的话，恢复到之前的状态，并报错。

注

可以将推送理解为远端仓库在 接受并应用 你本地仓库的提交历史

5、合并 (Merge)

合并是将一个分支的改动合并到另一个分支。 以一个合并提交的形式结束合并，合并提交会包含两个分支的所有改动。比如以下操作：

  1. 需求在开发分支完成开发，将该开发分支合并到主分支或者发布分支
  2. 修复 bug 时，将修复分支合并到主分支
  3. 版本正式发布以后，开始下一个版本的开发，现有版本的分支可以合并到主分支，以保持主分支的最新发育状态。
# 将feature分支合并到main
git checkout main
git merge feature/login

注

在合并的过程中，还涉及到 快进（Fast-forward） 的概念，是一种特殊的合并。为了避免混淆，不展开描述

6、变基 (Rebase)

变基是将一个分支的提交移动到另一个分支的顶部，使分支的提交历史更加线性。变基没有保留合并历史，更适合本地仓库里的分支操作

例如，本地临时创建了一个 temp 分支，修改完成后，将该分支的提交变基到版本分支，然后推送到远程 fork 仓库，fork 仓库通过 MR 的方式合并到主干仓库。

7、遴选 (Cherry-pick)

遴选是将某些提交应用到当前分支，他们可以来自任意分支。

例如，我们在开发分支上修复了一个 bug，主分支上也需要修复这个 bug，但是我们不想将开发分支上的历史全部合并过来，我们可以使用遴选来将该修复对应的提交单独应用到主分支。

合并和变基操作的是分支，遴选操作的是提交。

8、重置 (Reset)

Reset 可以将仓库重置到某个状态，可以放弃已提交的修改，也可以选择保留修改，将其恢复到暂存区。我们需要指定一个指针，来告诉 Reset 要将仓库重置到哪个状态。

一般根据参数来区分：

  • --soft 撤销：撤销提交，保留修改在暂存区。
  • --mixed 混合撤销：撤销提交，保留修改在工作区（默认）。
  • --hard 重置：撤销提交，丢弃所有修改。
git reset HEAD~1
git reset --soft HEAD~1  # 保留修改在暂存区
git reset --mixed HEAD~1 # 保留修改在工作区（默认）
git reset --hard HEAD~1  # 丢弃所有修改

9、撤销（Restore）

restore 就是撤销修改，主要用来将已暂存的修改恢复到工作区，或者将修改的文件恢复到修改之前的状态，也就是放弃修改。

10、撤回 (Revert)

撤回是使用一个新的提交来抵消掉目标提交，他的内容和目标提交的修改内容完全相反，从而使仓库回到该提交之前的状态。

11、储藏 (Stash)

储藏可以理解为是一个临时的抽屉，以提交（commit）的形式来临时保存工作目录的修改。储藏修改以后，工作区被清空，方便切换到其他分支或进行其他操作时，不出现冲突。在必要的时候，可以随时弹出储藏的修改，恢复到暂存区。

# 临时保存修改
git stash

# 恢复储藏
git stash pop

# 查看储藏列表
git stash list

12、子模块 (Submodule)

子模块也是一个 Git 仓库，作为另一个 Git 仓库的子目录。子模块可以用来管理项目的依赖，例如，一个项目依赖于另一个项目的库。

现在一般使用多个 Monorepo、Monorepo 架构来管理项目，而不是使用子模块。

# 添加子模块
git submodule add https://github.com/user/lib.git

# 更新子模块
git submodule update --init --recursive

13、冲突 (Conflict)

实质上，Git 的操作都是在操作提交历史，而不是操作文件。Git需要比较两个指针对应的状态之间的相对历史的差异。

  1. 不管是合并、变基还是遴选，Git 会先找到公共祖先，拿到公共祖先到两个指针之间的两段相对历史
  2. 将公共祖先到目标指针的这段历史里的提交，逐个与当前指针的相对历史进行合并
  3. 如果发现有冲突，Git 会停止比较，并提示用户解决冲突。用户需要手动编辑文件，解决冲突，然后标记为已解决。
  4. 继续比较下一个提交，直到比较完所有的提交。
  5. 如果没有冲突，Git 会将当前指针指向目标指针，完成合并。

忽略文件 (.gitignore)

忽略文件是用来告诉 Git 哪些文件不需要被版本控制。例如，编译生成的文件、临时文件、日志文件等。

# 忽略日志文件
*.log

# 忽略依赖目录
node_modules/
dist/

# 忽略环境配置文件
.env
*.env.local
